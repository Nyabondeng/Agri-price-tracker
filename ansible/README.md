# Ansible Configuration Management

This directory contains the Ansible playbook and inventory for automated deployment and configuration of the Agri Price Tracker application on Azure Virtual Machines.

## Overview

The Ansible playbook (`deploy.yml`) automates all server-side configuration tasks:

- Install Docker Engine and Docker Compose plugin
- Install system dependencies (curl, git, Python, etc.)
- Configure ACR (Azure Container Registry) authentication
- Pull the application Docker image from ACR
- Deploy and run the application container with proper configuration
- Handle service restarts and updates

## Architecture

```
GitHub Actions (CI/CD) 
         |
         v
   SSH Key Setup
         |
         v
   Generate Inventory from Secrets
         |
         v
   [Bastion Host] ← Jump server
         |
         v
   [Private App VM] ← Ansible target
         |
    Playbook Tasks:
    1. Install Docker
    2. Configure ACR auth
    3. Pull image
    4. Deploy container
    5. Verify health
         |
         v
   Application Running
```

## Prerequisites

### Development Machine
- **Ansible** >= 2.9 (install: `pip install ansible`)
- **Python** >= 3.8
- **SSH client** with key pair
- **jq** (optional, for JSON parsing)

### Azure Resources
- Application VM created by Terraform (see [../terraform/README.md](../terraform/README.md))
- Azure Bastion Host with public IP
- SSH key pair (generated by Terraform)
- ACR credentials for image authentication

### Connectivity
- You must reach the Bastion Host IP:22 from your local machine
- Ansible uses ProxyJump through Bastion to reach the private VM

## Installation

### 1. Install Ansible
```bash
# Using pip (recommended)
pip install ansible

# Or on macOS:
brew install ansible

# Verify installation:
ansible --version
```

### 2. Install Ansible Collections
```bash
cd ansible

# Install required collections
ansible-galaxy collection install community.docker
ansible-galaxy collection install ansible.posix

# Or use requirements:
ansible-galaxy install -r requirements.yml
```

### 3. Configure SSH
```bash
# Copy the Terraform-generated key
cp ../ssh-key-agric-price-tracker.pem ~/.ssh/
chmod 600 ~/.ssh/ssh-key-agric-price-tracker.pem

# Test SSH connectivity to Bastion
ssh -i ~/.ssh/ssh-key-agric-price-tracker.pem \
    azureuser@<BASTION_IP>

# Exit:
exit
```

## Inventory Configuration

### Manual Inventory Setup (for Local Testing)

Edit `inventory.ini` to target your VM:

```ini
[app]
# Direct access (if VM is public, which it shouldn't be)
app_vm ansible_host=<VM_IP> \
       ansible_user=azureuser \
       ansible_ssh_private_key_file=~/.ssh/ssh-key-agric-price-tracker.pem

[app:vars]
# Optional: Set variables for all hosts in [app]
ansible_python_interpreter=/usr/bin/python3
```

### Access Through Bastion (Recommended for Private VM)

Edit `inventory.ini` to use ProxyJump:

```ini
[app]
app_vm ansible_host=<PRIVATE_VM_IP> \
       ansible_user=azureuser \
       ansible_ssh_private_key_file=~/.ssh/ssh-key-agric-price-tracker.pem \
       ansible_ssh_common_args='-o ProxyJump=azureuser@<BASTION_IP>'

[app:vars]
ansible_python_interpreter=/usr/bin/python3
```

**Replace**:
- `<BASTION_IP>` — Public IP of Azure Bastion (from Terraform output `bastion_host_ip`)
- `<PRIVATE_VM_IP>` — Private IP of App VM (from Terraform output `app_vm_private_ip`)

### Automatic Inventory (CI/CD)

GitHub Actions CD pipeline automatically generates `inventory.ini` from secrets:

```bash
# CD workflow generates this automatically:
[app]
app_vm ansible_host=<DEPLOY_HOST> \
       ansible_user=azureuser \
       ansible_ssh_private_key_file=~/.ssh/id_rsa \
       ansible_ssh_common_args='-o ProxyJump=azureuser@<BASTION_HOST>'
```

## Playbook Variables

The `deploy.yml` playbook uses these variables (set via environment or `--extra-vars`):

### Application Variables
```yaml
app_dir: /opt/agri-price-tracker              # Where to deploy app
app_container_name: agric-price-tracker-app   # Docker container name
app_image: agri-price-tracker:latest          # Docker image (registry/name:tag)
app_port: 3000                                # Port app listens on inside container
```

### ACR Variables
```yaml
acr_login_server: <registry>.azurecr.io       # ACR URL
acr_username: <username>                      # ACR username
acr_password: <password>                      # ACR password
```

### Set Variables
```bash
# Via environment (used by CI/CD):
export APP_IMAGE="acragricpricetrickeravloqy.azurecr.io/agri-price-tracker:abc123def456"
export ACR_LOGIN_SERVER="acragricpricetrickeravloqy.azurecr.io"
export ACR_USERNAME="acradmin"
export ACR_PASSWORD="<password>"

# Or via command line:
ansible-playbook -i inventory.ini deploy.yml \
  -e "app_image=registry/image:tag" \
  -e "acr_login_server=registry.azurecr.io"

# Or in a vars file:
cat > vars.yml <<EOF
app_image: acragricpricetrickeravloqy.azurecr.io/agri-price-tracker:v1.0.0
acr_login_server: acragricpricetrickeravloqy.azurecr.io
acr_username: acradmin
acr_password: P@ssw0rd123
EOF

ansible-playbook -i inventory.ini deploy.yml -e @vars.yml
```

## Running the Playbook

### Syntax Check (No Changes)
```bash
# Verify YAML is valid and tasks are correct
ansible-playbook -i inventory.ini deploy.yml --syntax-check

# Output:
playbook: deploy.yml
```

### Dry Run (Predict Changes)
```bash
# Preview what will happen (doesn't make changes)
ansible-playbook -i inventory.ini deploy.yml --check

# Output shows tasks that WOULD run:
TASK [Update apt cache] ***
changed: [app_vm]
TASK [Install required OS packages] ***
changed: [app_vm]
...
```

### Actual Deployment
```bash
# Execute the playbook (makes real changes)
ansible-playbook -i inventory.ini deploy.yml

# With specific variables:
ansible-playbook -i inventory.ini deploy.yml \
  -e "app_image=registry/agri-price-tracker:sha-abc123" \
  -e "acr_login_server=r.azurecr.io"

# With extra verbosity (debug output):
ansible-playbook -i inventory.ini deploy.yml -vvv

# Limit to specific hosts:
ansible-playbook -i inventory.ini deploy.yml --limit app_vm
```

### Understanding Playbook Output

```
PLAY [Configure and deploy Agri Price Tracker] ****

TASK [Gathering Facts] ***
ok: [app_vm]

TASK [Update apt cache] ***
changed: [app_vm]

TASK [Install required OS packages] ***
changed: [app_vm]

TASK [Add Docker apt repository] ***
changed: [app_vm]

TASK [Install Docker engine and compose plugin] ***
changed: [app_vm]

TASK [Log in to Azure Container Registry] ***
ok: [app_vm]  # Skipped if ACR creds not provided

TASK [Pull application image] ***
changed: [app_vm]

TASK [Ensure app container is running] ***
changed: [app_vm]

PLAY RECAP ***
app_vm : ok=10  changed=8  unreachable=0  failed=0
```

**Result Codes**:
- `ok` — Task completed successfully, no changes
- `changed` — Task completed successfully, made changes
- `skipped` — Task was skipped (due to `when:` condition)
- `unreachable` — Cannot reach host (SSH problem)
- `failed` — Task failed

## Playbook Tasks Explained

### Task 1: Update apt cache
```yaml
- name: Update apt cache
  ansible.builtin.apt:
    update_cache: true
    cache_valid_time: 3600
```
Updates the Linux package list (required before installing packages).

### Task 2: Install OS packages
```yaml
- name: Install required OS packages
  ansible.builtin.apt:
    name:
      - ca-certificates
      - curl
      - gnupg
      - lsb-release
      - python3-pip
    state: present
```
Installs system dependencies needed for Docker installation.

### Task 3: Add Docker repository
```yaml
- name: Add Docker apt repository
  ansible.builtin.apt_repository:
    repo: "deb [arch=amd64 signed-by=/etc/apt/keyrings/docker.asc] https://download.docker.com/linux/ubuntu {{ ansible_distribution_release }} stable"
    state: present
```
Adds official Docker repository for latest versions.

### Task 4: Install Docker engine
```yaml
- name: Install Docker engine and compose plugin
  ansible.builtin.apt:
    update_cache: true
    name:
      - docker-ce
      - docker-ce-cli
      - containerd.io
      - docker-compose-plugin
    state: present
```
Installs Docker and Docker Compose plugin.

### Task 5: Start Docker service
```yaml
- name: Ensure Docker service is enabled and running
  ansible.builtin.service:
    name: docker
    state: started
    enabled: true
```
Starts Docker daemon and enables it to auto-start on reboot.

### Task 6: Log in to ACR
```yaml
- name: Log in to Azure Container Registry
  community.docker.docker_login:
    registry_url: "{{ acr_login_server }}"
    username: "{{ acr_username }}"
    password: "{{ acr_password }}"
  when:
    - acr_login_server | length > 0
```
Authenticates Docker with ACR using provided credentials (skipped if credentials not set).

### Task 7: Pull image
```yaml
- name: Pull application image
  community.docker.docker_image_pull:
    name: "{{ app_image }}"
```
Downloads the latest application Docker image from ACR.

### Task 8: Deploy container
```yaml
- name: Ensure app container is running
  community.docker.docker_container:
    name: "{{ app_container_name }}"
    image: "{{ app_image }}"
    state: started
    restart_policy: unless-stopped
    recreate: true
    pull: always
    published_ports:
      - "{{ app_port }}:3000"
    env:
      NODE_ENV: production
      PORT: "3000"
```
Creates or updates the Docker container with proper port mapping and environment variables.

## Troubleshooting

### Issue: SSH Connection Timeout
```
FAILED! — SSH: Get-Net-TCPConnection -LocalPort 22 timed out
```

**Solutions**:
1. Verify Bastion IP is correct and reachable
2. Check Azure NSG allows SSH from your IP
3. Verify SSH key file path is correct
4. Test SSH directly: `ssh -i ~/.ssh/key.pem azureuser@<BASTION_IP>`

### Issue: Docker Installation Fails

```
FAILED — Command 'docker --version' returned non-zero exit status 127
```

**Solutions**:
1. Run playbook again (may be transient)
2. Check VM has internet connectivity
3. Verify package manager cache isn't corrupted

### Issue: ACR Login Fails

```
FAILED — ... unauthorized: authentication required
```

**Solutions**:
1. Verify ACR credentials are correct
2. Check ACR_USERNAME and ACR_PASSWORD are set
3. Confirm credentials exist in Azure Portal → ACR → Access Keys
4. Verify image actually exists in ACR

### Issue: Image Pull Takes Too Long

**Solution**: Docker is pulling from network. Large images (>1GB) can take 5+ minutes. Be patient and check logs:

```bash
ansible all -i inventory.ini -m command -a "docker pull <image>" -vvv
```

### Issue: Container Exits After Starting

```bash
# Check container logs:
ansible all -i inventory.ini -m command -a "docker logs agric-price-tracker-app" -v
```

## Idempotency

The playbook is **idempotent** — running it multiple times produces the same result:

```bash
# First run: installs and configures everything
ansible-playbook -i inventory.ini deploy.yml

# Second run: verifies everything is correct, makes minimal changes
ansible-playbook -i inventory.ini deploy.yml

# Third run: same as second
ansible-playbook -i inventory.ini deploy.yml
```

This makes it safe to run the playbook on a schedule or after every code deployment.

## Advanced Usage

### Using Ansible Roles

For complex playbooks, organize tasks into roles:

```
ansible/
├── deploy.yml
├── roles/
│   ├── docker/
│   │   ├── tasks/main.yml
│   │   └── handlers/main.yml
│   └── app/
│       ├── tasks/main.yml
│       └── defaults/main.yml
```

Playbook:
```yaml
---
- name: Deploy Agri Price Tracker
  hosts: app
  become: true
  roles:
    - docker
    - app
```

### Using Handlers

Restart services only when configuration changes:

```yaml
tasks:
  - name: Copy app config
    copy:
      src: config.json
      dest: /opt/agri-price-tracker/config.json
    notify: Restart app container

handlers:
  - name: Restart app container
    command: docker restart agric-price-tracker-app
```

### Conditional Execution

Run tasks only under certain conditions:

```yaml
tasks:
  - name: Install on Ubuntu only
    apt:
      name: package
      state: present
    when: ansible_os_family == "Debian"

  - name: Pull new image if credentials provided
    docker_image_pull:
      name: "{{ app_image }}"
    when: acr_login_server is defined
```

## Best Practices

1. **Always run `--syntax-check` before deploying**: Catches YAML errors early
2. **Use `--check` mode first**: Preview changes before applying
3. **Add verbosity for debugging**: Use `-v`, `-vv`, or `-vvv`
4. **Store sensitive data in vault**: Use `ansible-vault` for secrets
5. **Test on staging first**: Don't deploy directly to production
6. **Document your variables**: Add comments explaining custom vars
7. **Use meaningful task names**: Makes logs easier to read
8. **Idempotent tasks only**: Ensure tasks produce same result on re-run
9. **Set appropriate delays**: Handle slow operations (e.g., apt updates)
10. **Monitor playbook runs**: Log output to file for audit trail

## Security Considerations

### Credential Management

**Don't**:
- Hardcode passwords in playbooks
- Store plain-text credentials in inventory.ini
- Commit SSH keys to repository

**Do**:
- Use Ansible Vault for secrets:
  ```bash
  ansible-vault create vars/secrets.yml
  ansible-playbook -i inventory.ini deploy.yml --ask-vault-pass
  ```
- Use environment variables (stored in GitHub Secrets)
- Use Azure RBAC and managed identity
- Restrict SSH key permissions (0600)

### Network Security

- Bastion Host limits SSH access
- Application VM has no direct internet access
- Database only accepts connections from App VM
- Use SSH keys, not passwords

### Container Security

- Use minimal base images
- Pin image versions (not `latest`)
- Scan images for vulnerabilities (done in CI)
- Run containers as non-root user (when possible)
